# Firebase 即時同步手寫白板開發規格書

## 1. 專案基本資訊

### 1.1 專案名稱

```text
firebase-live-whiteboard
```

### 1.2 專案定位

本專案是一套針對 iPad 與觸控裝置優化的課堂即時同步白板。

老師不需要註冊或輸入帳號密碼，即可建立一個臨時課堂、貼上學生名單，並產生每位學生專屬的白板網址。

學生透過專屬網址進入白板後，可使用手指或 Apple Pencil 書寫。老師可在監看頁面即時查看學生作答，並放大單一學生的白板進行批注。

學生筆跡與老師批注必須分別儲存在不同資料節點與不同畫布圖層。學生只能編輯自己的筆跡，不能修改老師批注。

### 1.3 部署方式

* 前端託管：GitHub Pages。
* 資料庫：Firebase Realtime Database。
* 身分識別：Firebase Anonymous Authentication。
* 後端：不使用 Cloud Functions。
* 檔案儲存：不使用 Firebase Storage。
* Firebase 方案：Spark 免費方案。
* 圖片匯出：瀏覽器本機產生並下載。
* 開發工具：Codex。
* 程式語言：HTML、CSS、JavaScript ES Modules。
* 建置工具：Vite。
* Firebase SDK：npm 安裝的 Firebase Modular Web SDK。

Firebase 官方建議現代 JavaScript 專案透過 npm 與模組化 SDK 使用 Firebase，以利模組打包與移除未使用的程式碼。

# 2. 核心限制

## 2.1 不使用老師正式登入

老師端不提供：

* Google 登入。
* Email 登入。
* 密碼登入。
* 老師帳號註冊。
* Email 找回。
* 跨裝置帳號同步。

老師第一次建立課堂時，系統在背景使用 Firebase Anonymous Authentication 取得匿名 UID。

老師不會看到登入畫面。Firebase Authentication 必須使用瀏覽器本機持久化；建立課堂後，前端另以 `localStorage` 暫存課堂 ID、老師 UID 與時間，不保存六位數密鑰。跳轉管理頁時必須先等待本機匿名身分恢復完成，確認 UID 後再讀取課堂權限，避免同一瀏覽器被誤判為未授權裝置。

## 2.2 老師連結、六位數密鑰與跨裝置管理

課堂建立後，將建立課堂的匿名 UID 設為該課堂管理員。

```text
classes/{classId}/admins/{teacherUid} = true
```

課堂建立後，系統同時建立永久老師連結及六位數字密鑰。管理頁直接顯示完整密鑰，不使用密碼遮蔽。新瀏覽器開啟老師連結時會看到密鑰輸入表單；輸入正確密鑰後，其匿名 UID 會被加入同一課堂的 `admins`，並與該課堂其他老師享有相同完整權限，不限連線數量。

六位數密鑰存放於受規則保護的 `teacherKeys/{classId}`；未授權瀏覽器不能讀取，只能透過 `teacherKeyClaims/{classId}/{uid}` 提交相符密鑰。錯誤密鑰不得建立 claim，也不得加入 `admins`。

下列操作可能造成老師失去管理權限：

* 清除瀏覽器網站資料。
* 清除 Safari 或 Chrome 儲存空間。
* 使用無痕模式。
* 移除 Firebase 匿名帳號。
* 使用另一個 GitHub Pages 網域。

只要保留老師連結與六位數密鑰，即可在新瀏覽器或新裝置重新登入；若遺失兩者，系統不提供人工找回。

## 2.3 老師登入資料提示

介面必須顯示：

> 此課堂的老師連結與六位數密鑰可供其他裝置登入。所有已登入的老師裝置享有相同完整權限，請只提供給授課老師。

## 2.4 學生第一次使用後綁定裝置

每位學生取得一個高強度隨機網址。

學生不需要也不得輸入老師六位數密鑰；學生只需開啟完整的個人專屬網址，即可在首次使用時綁定目前裝置。

學生第一次打開網址時：

1. 背景建立匿名 Firebase 身分。
2. 檢查白板是否尚未綁定學生 UID。
3. 若尚未綁定，將目前匿名 UID 寫入 `studentUid`。
4. 後續只有相同 UID 可以修改學生圖層。

這表示學生網址在第一次使用前必須妥善保管。若其他人先開啟該網址，可能先取得該白板的編輯權限。

老師端必須提供「解除學生裝置綁定」功能。

* 查看其他學生的白板。

* 修改老師批注。
* 刪除老師批注。
* 將自己加入課堂管理員。
* 修改課堂設定。
* 修改自己的 `studentUid`。
* 修改學生姓名或座號。
* 在白板被鎖定後繼續寫入。

# 4. 系統頁面

## 4.1 首頁

檔案：

```text
index.html
```

功能：

* 顯示系統名稱與簡介。
* 建立新課堂。
* 以「我的課堂（只會顯示您這個瀏覽器建立的課程）」標示課堂清單。
* 顯示目前瀏覽器建立過的課堂。
* 測試階段另以「所有課程（只有現在測試階段時可見）」區塊列出系統上的全部課程，並標示「進行中」或「停用中」及距離自動刪除的剩餘時間；首頁需定期更新倒數與狀態。每列提供老師連結；六位數密鑰只對目前匿名 UID 已有管理權限的課程顯示，其他課程不得放寬規則讀取。此清單只讀取不含管理員 UID、學生索引或課堂設定的精簡公開課程目錄。
* 在隱私提醒上方只統計系統目前進行中的課程數量。
* 進入既有課堂管理頁。
* 使用老師端前端時，自動清除建立已滿 3 小時的課程。
* 顯示使用限制與隱私說明。
* 顯示資料庫連線狀態。
* 提醒老師妥善保存課程網址與六位數密鑰，遺失兩者將無法進入課程。
* 提供「清除瀏覽記錄」操作，經確認後清除本網站的全部 `localStorage`。

首頁應使用目前匿名 UID 查詢：

```text
userClasses/{auth.uid}
```

並顯示該瀏覽器曾建立或被授權管理的課堂。

課程統計直接依精簡公開課程目錄計算狀態為進行中且尚未到期的課程，不在瀏覽器保存歷史建立或刪除紀錄。刪除課程時，課堂主資料、關聯資料與公開課程目錄必須以同一筆 Firebase 原子更新刪除，不留下課程紀錄。

清除瀏覽記錄前必須提醒老師先保存課程網址與六位數密鑰。此操作清除以 `classpad` 開頭的老師登入暫存與介面偏好，但不得刪除 Firebase 匿名驗證權杖，也不得刪除 Firebase 中的課堂、學生或筆跡資料。建立新課堂前必須強制刷新匿名驗證 token；若刷新時或資料庫建立操作回覆 token 已失效，則重新建立匿名身分、重啟 Realtime Database 連線，並只重試建立一次。

## 4.2 老師設定頁

檔案：

```text
teacher.html
```

網址：

```text
teacher.html?class={classId}
```

功能區塊：

### 課堂資料

* 課堂名稱。
* 班級名稱。
* 活動名稱。
* 建立時間。
* 最後更新時間。
* 距離建立滿 3 小時自動清除的剩餘小時與分鐘。
* 課堂狀態。
* 是否允許學生書寫。
* 是否允許學生自行清除。
* 是否顯示老師批注。

上述課堂資訊與老師連結 QR、展示畫面 QR、下載連結 CSV、下載全班作品 ZIP、結束課堂、刪除課堂六個操作按鈕，應依序集中於頁面最上方的固定工具列。

### 學生名單輸入

老師可貼入：

```text
01 王小明
02 陳小華
03 林小美
```

也可貼入：

```text
王小明
陳小華
林小美
```

解析規則：

* 忽略空白行。
* 去除行首與行尾空白。
* 若第一段為數字，視為座號。
* 若沒有座號，自動依序編號。
* 相同姓名可以存在。
* 座號可以包含英文字母。
* 匯入內容中或現有名單內若已有完全相同的「座號＋姓名」，應略過且不重複新增。
* 每位學生都產生獨立的 `studentId` 與 `boardToken`。
* 每班最多 80 人。
* 匯入區提供「填入 10 人範例」按鈕，按下後只將十組座號與姓名填入輸入框，由老師確認後再加入名單。

### 學生清單

每一列顯示：

* 座號。
* 姓名。
* 連線狀態。
* 是否綁定裝置。
* 是否鎖定。
* 複製網址。
* 顯示 QR Code。
* 解除裝置綁定。
* 重新產生網址。
* 清除白板。
* 刪除學生。

# 5. 白板畫布設計

## 5.1 畫布圖層

畫面使用三個重疊 Canvas：

學生白板、老師批注畫面與多格縮圖統一固定為 4:3 長寬比。

```html
<div class="board-stage">
  <canvas id="backgroundCanvas"></canvas>
  <canvas id="studentCanvas"></canvas>
  <canvas id="teacherCanvas"></canvas>
</div>
```

圖層順序：

1. 背景圖層。
2. 學生筆跡圖層。
3. 老師批注圖層。

學生端：

* `studentCanvas` 接收書寫事件。
* `teacherCanvas` 設為唯讀。
* 老師圖層必須設置 `pointer-events: none`。

老師端：

* 學生圖層唯讀。
* 老師圖層接收書寫事件。

## 5.2 畫布邏輯尺寸

使用固定邏輯尺寸：

```text
1600 × 1200
```

資料庫保存正規化座標：

```text
x = logicalX / 1600
y = logicalY / 1200
```

座標保存範圍：

```text
0.0000 ～ 1.0000
```

畫面顯示時，再依目前畫布尺寸換算。

## 5.3 高解析度顯示

Canvas 實際像素尺寸需乘上：

```javascript
window.devicePixelRatio
```

但 Canvas 的 CSS 寬高維持畫面尺寸，避免 Retina 螢幕筆跡模糊。

## 5.4 Pointer Events

輸入事件統一使用：

* `pointerdown`
* `pointermove`
* `pointerup`
* `pointercancel`

記錄：

* `pointerId`
* `pointerType`
* `pressure`
* `clientX`
* `clientY`
* `timeStamp`

## 5.5 iPad 手勢

操作原則：

* 一個觸控點：書寫。
* Apple Pencil：書寫。
* 兩個觸控點：縮放與拖曳。
* 雙指操作開始時，取消尚未完成的單指筆畫。
* 雙指操作不產生筆跡。
* 縮放範圍為 25%～400%。
* 提供重設為 100% 按鈕。

畫布容器 CSS：

```css
.board-stage {
  touch-action: none;
  overscroll-behavior: none;
  user-select: none;
  -webkit-user-select: none;
  -webkit-touch-callout: none;
}
```

## 5.6 Apple Pencil 壓力

若 `event.pressure` 可用，筆畫寬度可依壓力調整：

```text
renderWidth = baseWidth × (0.6 + pressure × 0.8)
```

若壓力值為 0 或瀏覽器不支援，使用固定寬度。

# 6. 筆跡格式

## 6.1 完整筆畫

```json
{
  "id": "stroke_x7Jp29",
  "tool": "pen",
  "color": "#1f2937",
  "width": 4,
  "opacity": 1,
  "points": [
    [0.1523, 0.2814, 0.52, 0],
    [0.1541, 0.2840, 0.56, 16],
    [0.1570, 0.2879, 0.61, 33]
  ],
  "createdAt": 1784421000000,
  "authorUid": "firebase-anonymous-uid"
}
```

點陣列內容：

```text
[x, y, pressure, elapsedMilliseconds]
```

## 6.2 資料限制

* 單筆最多 500 個點。
* 超過 500 點時切割為下一筆。
* 座標最多保留四位小數。
* 壓力最多保留兩位小數。
* 顏色只接受 `#RRGGBB`。
* 寬度限制 1～40。
* 透明度限制 0.1～1。
* `tool` 第一版只接受 `pen`。
* 文字欄位不得包含 HTML。
* `authorUid` 必須等於目前登入 UID。

## 6.3 路徑簡化

完成一筆時：

1. 移除距離小於指定閾值的連續點。
2. 對長路徑套用簡化演算法。
3. 限制最大點數。
4. 寫入完整筆畫。
5. 清除即時暫存筆畫。

# 7. 橡皮擦

## 7.1 第一版採整筆擦除

橡皮擦碰觸一筆筆畫時，刪除整個 stroke。

需要建立筆畫命中測試：

```text
distance(pointer, strokeSegment) <= eraserRadius
```

## 7.2 權限

學生橡皮擦只能刪除：

```text
studentStrokes
```

老師橡皮擦只能刪除：

```text
teacherStrokes
```

老師若要清除學生筆跡，必須使用管理介面中的「清除學生白板」，不能透過老師橡皮擦混合操作。

# 8. 復原與重做

## 8.1 操作紀錄

每個裝置維護本機歷史堆疊：

```javascript
undoStack
redoStack
```

可復原操作：

* 新增筆畫。
* 刪除筆畫。
* 清除自己的圖層。

## 8.2 範圍

學生只能復原目前裝置在本次頁面工作階段所做的操作。

老師只能復原老師圖層操作。

第一版不提供跨裝置、跨重新整理的完整復原歷史。

# 9. 即時同步策略

## 9.1 本機優先繪製

使用者書寫時，筆跡先直接繪製於本機 Canvas，不等待 Firebase 回應。

這能避免網路延遲造成書寫卡頓。

## 9.2 書寫中的暫存筆畫

書寫期間每隔約 100 毫秒更新：

```text
activeStrokes/{boardToken}/{layer}/{uid}
```

資料包含：

* 當前 stroke ID。
* 顏色。
* 寬度。
* 最近一批座標。
* 更新時間。

## 9.3 完成筆畫

抬筆後：

1. 將完整筆畫寫入 `studentStrokes` 或 `teacherStrokes`。
2. 寫入成功後刪除 `activeStrokes`。
3. 更新 `updatedAt`。
4. 增加 `revision`。

## 9.4 Realtime Database 監聽

監聽新增與刪除事件：

* `onChildAdded`
* `onChildRemoved`
* `onValue`

避免每次變動都重新下載整個白板。

Realtime Database 的 Web SDK 可讓不同客戶端共享資料庫實例並即時接收最新資料。

## 9.5 離線狀態

使用：

```text
/.info/connected
```

判斷資料庫連線狀態，並顯示：

* 已同步。
* 同步中。
* 離線。
* 同步失敗。

Realtime Database 官方提供 `/.info/connected` 供 Web 應用判斷連線狀態。

# 13. Firebase Security Rules

## 13.1 基本原則

Realtime Database Security Rules 會在 Firebase 伺服器端強制執行，並可依 Firebase Authentication 的 `auth.uid` 控制個別使用者的讀寫權限。預設規則不應開放所有人讀寫。

禁止使用：

```json
{
  "rules": {
    ".read": true,
    ".write": true
  }
}
```

## 13.2 建議規則檔

檔案：

```text
database.rules.json
```

初版規則：

# 17. 隨機識別碼

## 17.1 課堂 ID

格式：

```text
8～12 位英數字
```

使用：

```javascript
crypto.getRandomValues()
```

不可使用：

* `Math.random()`。
* 日期加流水號。
* 可猜測的班級名稱。
* 連續課堂編號。

## 17.2 Board Token

至少使用 128-bit 隨機資料，轉成 URL-safe 字串。

範例：

```text
Qx7P9m2VfK8wT4nRz6Bc
```

## 17.3 Stroke ID

格式：

```text
stroke_{timestamp}_{random}
```

或使用 UUID。

# 20. 本機 PNG 匯出

老師在批注模式點擊「下載 PNG」時：

1. 建立離線合成 Canvas。
2. 畫入背景。
3. 畫入學生圖層。
4. 畫入老師批注圖層。
5. 在圖片邊緣加入姓名、座號與匯出時間。
6. 使用 `canvas.toBlob()`。
7. 建立臨時下載連結。
8. 下載完成後撤銷 Object URL。

檔名：

```text
座號-姓名-YYYYMMDD-HHmm.png
```

不得將 PNG 上傳至 Firebase。

## 20.1 全班作品 ZIP 匯出

老師可在課堂管理工具列點擊「下載全班作品 ZIP」。系統逐位學生、逐頁讀取筆跡，在瀏覽器本機將學生筆跡畫在下層、老師筆跡畫在上層，並將每個有筆跡的頁面輸出為獨立 PNG。沒有任何學生或老師筆跡的空白頁不輸出；完全沒有筆跡的學生不建立資料夾。

ZIP 內容依「座號-姓名」建立學生資料夾，圖片以頁次命名。ZIP 必須包含 UTF-8 編碼的 `無筆記學生.txt`，列出完全沒有筆跡的學生；若所有學生都有筆記，文字檔註明「所有學生皆有筆記。」。匯出期間按鈕顯示處理進度並停止重複操作。為避免 iPad Safari 記憶體不足，依序產生圖片，PNG 總大小上限為 250 MB。所有合成與封裝只在瀏覽器本機進行，不上傳 PNG 或 ZIP。

# 23. Codex 開發規則

```text

```

內容要求：

```text
# Codex Development Instructions

## General

- Read SPEC.md before modifying any code.
- Implement the project incrementally.
- Do not introduce a backend server.
- Do not use Firebase Cloud Functions.
- Do not use Firebase Storage.
- Do not require teacher or student account registration.
- Use Firebase Anonymous Authentication in the background.
- Use Firebase Realtime Database only.
- Deploy the frontend to GitHub Pages.
- Keep the project compatible with the Firebase Spark free plan.

## Technology

- Use Vite.
- Use plain JavaScript ES Modules.
- Use the Firebase modular Web SDK.
- Do not use React, Vue, Angular, or other UI frameworks unless explicitly requested.
- Do not use external APIs for QR codes or image export.
- Use Pointer Events for mouse, touch, and Apple Pencil.
- Optimize primarily for iPad Safari.

## Security

- Never use open Firebase rules.
- Never commit a service account key.
- Never use Firebase Admin SDK in frontend code.
- Do not trust classId, boardToken, studentId, or UID values supplied by the browser.
- Validate all writable fields in database.rules.json.
- Students must never write teacherStrokes.
- Students must never add themselves to classes/{classId}/admins.
- Teachers must not edit teacher layers of classes they do not administer.
- Add Emulator tests for allowed and denied operations.

## Canvas

- Use separate background, student, and teacher Canvas layers.
- Store coordinates in normalized 0–1 format.
- Render Canvas at devicePixelRatio.
- One pointer or Apple Pencil draws.
- Two touch pointers pan and zoom.
- Cancel the current stroke when a two-finger gesture begins.
- Keep the teacher layer read-only on the student page.
- Keep the student layer read-only in teacher review mode.

## Firebase synchronization

- Draw locally before writing to Firebase.
- Throttle active stroke updates.
- Store completed strokes as one object.
- Use onChildAdded and onChildRemoved instead of repeatedly reading the entire board.
- Remove listeners when changing monitor pages or closing review mode.
- Use onDisconnect for presence cleanup.

## Quality

- Run npm test and npm run build after each major task.
- Do not leave placeholder buttons.
- Handle loading, empty, offline, permission-denied, and invalid-link states.
- Keep UI text in Traditional Chinese.
- Ensure interactive controls have accessible labels.
- Avoid horizontal scrolling on iPad.
```
